2.1.8. wff_language.parse_tree module¶

The parse tree for the language.

Should the nodes of the tree be generic nodes with a type field or get/set methods, or should we derive a special node for each type of grammar element quantifiers, stdFunctions, variables, etc.?

The derived class thing is nice because the different language elements might need specialized member functions and data stored with them. But the simplicity of only having a single type of tree node is also appealing, not having to bother with subclasses and superclasses.

Recall that generally an AST for a parsed language has a node for each distinct lhs of a production in the grammar of the language, as well as for each literal (a literal is a quoted sequence of actual characters in the language). A depth-first search (DFS) of the tree, printing literals associated with each node, should reproduce the original string which was parsed.

In our case, we are dealing with wffs and meta-wffs of logic as the primary object of interest, but these are elements of a larger language which includes, for example, commands to define new variable names and function names.
